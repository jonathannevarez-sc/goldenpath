# What Is Golden Path?

## A simple guide  
**Platform version:** v0.3.7 · **Date:** 2026-06-24

---

## In plain English

**Golden Path is your organization’s standard way to build and launch cloud software.**

Instead of every team figuring out deployment, security, and monitoring on their own, Golden Path gives them a **proven starting point** and an **automated path to production**. Developers spend time on product features. The platform handles the repeatable infrastructure work.

Think of it as a **factory line for cloud apps**: the conveyor belt, safety checks, and quality standards are built once. Each new product rolls off the line with the same structure — faster, safer, and easier to support.

---

## The business problem

Most organizations pay the same setup cost again and again for every new service:

| What happens today | What it costs you |
|--------------------|-------------------|
| Each team designs its own deploy process | Weeks before the first feature ships |
| Security and monitoring added late (or skipped) | Higher incident and compliance risk |
| Best practices live in wikis and people’s heads | Knowledge drifts; onboarding stays slow |
| Every service looks different to operations | Expensive support; hard to audit |

**Golden Path turns that repeated cost into a shared asset** — one platform investment that every new service benefits from.

---

## What Golden Path does

Golden Path is a **shared platform repository** (not a single application). Your organization configures it once for your cloud account and code hosting. Then:

1. **A developer picks a starting template** — web app, API, dashboard, or similar (six options today).
2. **A complete project is created automatically** — including security, hosting, and deploy automation.
3. **The developer writes product code** and pushes changes.
4. **The service deploys itself** to your cloud environment — development first, production when ready.

No manual cloud console work after the initial setup. No long-lived passwords stored in code.

---

## How it helps your organization

### For the business

- **Faster time to market** — target: first live development environment in under one business day.
- **Lower cost per new service** — setup work is done once, not reinvented per team.
- **Predictable risk** — security and monitoring are built in from day one, not bolted on later.
- **Easier audits** — every service follows the same recognizable pattern.

### For engineering leadership

- **Consistent delivery** — teams produce services that look and behave the same way.
- **Less platform distraction** — product engineers focus on features, not infrastructure plumbing.
- **One thing to support** — platform and operations teams learn one model, not dozens.

### For security and compliance

- **Guardrails by default** — secrets in a vault, least-privilege access, keyless automation.
- **Controlled changes** — production actions require human confirmation; sensitive steps are logged.
- **No shadow runbooks** — official guidance is served from one maintained source (including AI assistants).

### For executives

- **Pilot-ready today** — the paved road and AI front desk are built; you can sponsor a small team to prove ROI.
- **Opt-in, not forced** — existing systems can stay as-is; Golden Path targets new services and willing adopters.
- **Business continuity** — core deploys work even if the AI help desk is temporarily unavailable.

## How teams actually use it

Teams can start three ways — all leading to the same result:

| Path | Who it suits |
|------|--------------|
| **Guided wizard** | Teams that want step-by-step setup without deep technical skill |
| **Command-line tools** | Engineers comfortable in the terminal |
| **AI assistant (MCP)** | Teams using Claude or similar tools — official playbooks and status checks in one place |

Regardless of path, the outcome is the same: a **service repository** on your code platform that deploys automatically to **Google Cloud**.

---

## The simple mental model

```
Official guidance  →  Create from template  →  Push code  →  Automated deploy  →  Live service
     (optional AI)         (minutes)              (daily)        (no manual steps)
```

**Three parts to remember:**

| Part | Role |
|------|------|
| **Templates & automation** | The factory — how services are built and deployed |
| **AI front desk (MCP)** | The help desk — official docs, playbooks, and status checks |
| **Google Cloud** | Where applications actually run |

The help desk **assists** people. It does **not** replace the factory. If the help desk is down, teams still deploy by pushing code — the business is not blocked.

## What Golden Path is not

| Misconception | Reality |
|---------------|---------|
| A mandate to migrate everything | Opt-in; legacy systems can remain |
| AI running production alone | Humans own production; AI assists within guardrails |
| One tool for every possible architecture | Covers the **common case** first; special cases need platform consultation |
| A replacement for your cloud or code platform | It sits on top of Google Cloud and GitHub — it organizes how you use them |
